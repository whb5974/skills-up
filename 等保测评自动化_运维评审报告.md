## 一、部署方案优点

### 1.1 架构分层清晰

方案采用了标准的四层架构设计（用户层→应用层→工具层→数据层），层次分明，符合运维最佳实践：

- **用户层**支持多种接入方式（Web、飞书机器人、API、命令行），便于不同场景使用
- **应用层**按测评流程四阶段划分模块，业务逻辑清晰
- **工具层**集成成熟开源工具（OpenVAS、OpenSCAP、Nmap），降低自研风险
- **数据层**采用飞书多维表格 + MySQL 混合存储，兼顾灵活性与性能

### 1.2 技术选型合理

- **FastAPI**：高性能异步框架，适合 I/O 密集型的扫描任务
- **Celery + Redis**：成熟的任务队列方案，支持分布式任务调度
- **MySQL 8.0**：稳定可靠，支持复杂查询和事务
- **开源扫描工具**：OpenVAS、OpenSCAP、Nmap 均为业界标准工具，社区活跃

### 1.3 与标准流程对应良好

方案严格遵循 GB/T 28449-2018 四阶段流程，每个阶段都有明确的交付物和自动化支持，便于运维人员理解业务逻辑。

## 二、运维风险（至少 5 项）

### ⚠️ 风险 1：数据库单点故障风险

**问题描述：**
方案中数据库采用单一 MySQL 实例，未设计主备复制或集群方案。测评结果数据库存储所有测评数据，一旦数据库故障，将导致：
- 正在执行的测评任务中断
- 历史测评数据无法访问
- 报告生成失败

**影响等级：** 🔴 高  
**发生概率：** 中  

**建议措施：**
- 部署 MySQL 主从复制（至少 1 主 1 从）
- 配置半同步复制，确保数据一致性
- 考虑使用 MGR（MySQL Group Replication）或 PXC 集群
- 部署数据库监控，监控主从延迟、连接数、慢查询

---

### ⚠️ 风险 2：扫描引擎资源竞争与隔离不足

**问题描述：**
方案中 OpenVAS、OpenSCAP、Nmap 等扫描工具直接部署在应用服务器或同一网络环境，存在以下风险：
- 多个扫描任务并发执行时，CPU/内存资源竞争严重
- 扫描流量可能影响被测系统正常运行
- 扫描引擎故障可能级联影响应用服务

**影响等级：** 🟠 中高  
**发生概率：** 高  

**建议措施：**
- 扫描引擎独立部署，与应用服务器物理或逻辑隔离
- 使用 Docker 容器化部署扫描工具，实现资源限制和隔离
- 配置扫描任务速率限制，避免网络风暴
- 建立扫描引擎健康检查机制

---

### ⚠️ 风险 3：监控告警设计缺失

**问题描述：**
方案中仅在"交付物清单"中提到"监控告警系统"，但无具体设计：
- 未定义监控指标（CPU、内存、磁盘、网络、服务状态）
- 未设计告警阈值和告警级别
- 未明确告警通知渠道和升级机制
- 无日志集中收集和分析方案

**影响等级：** 🟠 中高  
**发生概率：** 高  

**建议措施：**
- 部署 Prometheus + Grafana 监控体系
- 定义关键监控指标和告警阈值（见后文补充）
- 集成飞书告警机器人，实现告警通知
- 部署 ELK 或 Loki 日志系统，集中收集应用日志和系统日志

---

### ⚠️ 风险 4：备份恢复策略不完整

**问题描述：**
方案中未提及数据备份和灾难恢复设计：
- 数据库备份策略（全量/增量、备份频率、保留周期）未定义
- 配置文件、脚本、报告模板的备份方案缺失
- 未设计灾难恢复演练计划
- 备份数据的有效性验证机制缺失

**影响等级：** 🟠 中高  
**发生概率：** 中  

**建议措施：**
- 数据库每日增量备份，每周全量备份
- 备份数据异地存储（至少不同物理位置）
- 定期（每季度）进行备份恢复演练
- 监控备份任务执行状态，失败时告警

---

### ⚠️ 风险 5：权限控制设计过于简化

**问题描述：**
方案中提到"人员管理模块"，但权限控制设计过于简化：
- 未定义角色权限模型（RBAC）
- 扫描引擎执行权限未隔离（可能以 root 运行）
- 数据库访问权限未细化（应用账号权限过大）
- 未设计操作审计日志

**影响等级：** 🟠 中高  
**发生概率：** 中  

**建议措施：**
- 实现基于角色的访问控制（RBAC）
- 扫描任务使用最小权限原则运行
- 数据库账号按功能分离（只读、读写、管理）
- 记录所有关键操作的审计日志（谁、何时、做了什么）

---

### ⚠️ 风险 6：高可用性设计缺失

**问题描述：**
方案中未提及任何高可用性设计：
- 应用服务单点部署，故障后无法自动切换
- 未设计负载均衡方案
- 无故障自动检测和恢复机制
- 未定义 RTO（恢复时间目标）和 RPO（恢复点目标）

**影响等级：** 🟡 中  
**发生概率：** 中  

**建议措施：**
- 应用服务至少部署 2 实例，前端配置负载均衡
- 使用 Keepalived 或云厂商负载均衡器实现故障切换
- 定义 RTO ≤ 30 分钟，RPO ≤ 15 分钟
- 编写故障恢复手册，定期演练

---

### ⚠️ 风险 7：漏洞扫描器自身安全风险

**问题描述：**
OpenVAS 等漏洞扫描器本身存在安全风险：
- 漏洞库需要定期更新，更新失败可能导致漏报
- 扫描器可能被攻击者利用进行未授权扫描
- 扫描结果包含敏感信息，泄露风险高

**影响等级：** 🟡 中  
**发生概率：** 低  

**建议措施：**
- 配置漏洞库自动更新，监控更新状态
- 扫描器部署在独立安全域，限制访问来源
- 扫描结果加密存储，访问需要审批
- 定期评估扫描器自身安全性

### 3.4 容器化部署建议

使用 Docker Compose 或 Kubernetes 部署，优势：
- 环境一致性，避免"在我机器上能跑"问题
- 资源隔离和限制，避免扫描任务耗尽资源
- 快速扩缩容，应对并发测评需求
- 便于版本管理和回滚

示例 Docker Compose 结构：
```yaml
version: '3.8'
services:
  app:
    image: etcd-assessment:latest
    deploy:
      replicas: 3
      resources:
        limits:
          cpus: '2'
          memory: 4G
  
  celery-worker:
    image: etcd-assessment:latest
    command: celery -A app worker -l info
    deploy:
      replicas: 5
  
  redis:
    image: redis:7-alpine
    deploy:
      replicas: 3  # Sentinel 模式
  
  mysql:
    image: mysql:8.0
    deploy:
      replicas: 1  # 主库
    volumes:
      - mysql-data:/var/lib/mysql
  
  mysql-slave:
    image: mysql:8.0
    deploy:
      replicas: 2  # 从库
```

| 类别 | 监控指标 | 告警阈值 | 告警级别 |

| | 内存使用率 | > 85% 持续 5 分钟 | 🟠 警告 |
| | 磁盘使用率 | > 80% | 🟠 警告 |

- **方案 A（轻量）**：Loki + Promtail + Grafana
- **方案 B（完整）**：Elasticsearch + Logstash + Kibana
- **方案 C（云原生）**：云厂商日志服务（阿里云 SLS、腾讯云 CLS）

### 4.3 备份恢复流程

#### 备份策略

| 数据类型 | 备份方式 | 频率 | 保留周期 | 存储位置 |
|----------|----------|------|----------|----------|
| MySQL 数据库 | 全量 + 增量 | 周日全量，每日增量 | 30 天 | 本地 + 异地 |
| Redis 数据 | RDB 快照 | 每小时 | 7 天 | 本地 |
| 配置文件 | 版本控制 + 备份 | 变更时 | 永久 | Git + 备份服务器 |
| 扫描结果 | 归档备份 | 每周 | 365 天 | 对象存储 |
| 报告文件 | 归档备份 | 每日 | 365 天 | 对象存储 |

#### 恢复演练计划

- **频率**：每季度至少 1 次
- **范围**：数据库恢复、应用恢复、全系统恢复
- **记录**：记录恢复时间、问题、改进措施
- **目标**：RTO ≤ 30 分钟，RPO ≤ 15 分钟

### 4.5 安全运维设计

#### 权限控制模型（RBAC）

| 角色 | 权限范围 | 操作限制 |
|------|----------|----------|
| 系统管理员 | 系统配置、用户管理、备份恢复 | 不能查看测评结果 |
| 测评管理员 | 项目管理、方案审批、报告审批 | 不能修改系统配置 |
| 测评工程师 | 执行测评、查看结果、生成报告 | 不能审批、不能删除数据 |
| 审计员 | 查看日志、查看审计记录 | 只读权限 |
| 只读用户 | 查看报告、查看统计 | 只读权限 |

#### 安全加固建议

1. **系统层面**
   - 关闭不必要的系统服务
   - 配置防火墙，仅开放必要端口
   - 定期更新系统补丁
   - 配置 SSH 密钥登录，禁用密码登录

2. **应用层面**
   - 启用 HTTPS，配置 TLS 1.2+
   - 配置 API 访问频率限制
   - 敏感操作需要二次确认
   - 会话超时时间 ≤ 30 分钟

3. **数据库层面**
   - 应用账号最小权限原则
   - 禁止远程 root 登录
   - 启用 SQL 审计日志
   - 敏感数据加密存储

4. **网络层面**
   - 扫描引擎独立网段
   - 数据库不直接暴露公网
   - 配置网络 ACL，限制访问来源
   - 扫描流量限速，避免影响业务

|----------|------|------|
| 业务功能设计 | ⭐⭐⭐⭐ | 流程完整，自动化覆盖合理 |
| 技术架构设计 | ⭐⭐⭐ | 分层清晰，但高可用缺失 |
| 部署架构设计 | ⭐⭐ | 单点故障风险高 |
| 监控告警设计 | ⭐ | 基本缺失 |
| 安全运维设计 | ⭐⭐ | 权限控制过于简化 |
| 运维手册完整性 | ⭐⭐ | 需要大量补充 |

### 5.2 核心建议

1. **优先解决单点故障**：数据库主从复制、应用多实例部署
2. **补充监控告警体系**：部署 Prometheus + Grafana，定义关键指标
3. **完善备份恢复策略**：制定备份计划，定期演练
4. **强化安全运维设计**：实现 RBAC 权限模型，配置审计日志
5. **编写完整运维手册**：包含部署、监控、故障处理全流程
